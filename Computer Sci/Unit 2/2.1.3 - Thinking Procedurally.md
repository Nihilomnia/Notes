## Identify the components of a problem

In computer science, thinking procedurally makes the task of writing a program a lot simpler by breaking a problem down into smaller parts which are easier to understand and consequently, easier to design.

The first stage of thinking procedurally in software development involves taking the problem defined by the user and breaking it down into its component parts, in a process called problem decomposition. In this process, a **large, complex problem is continually broken into smaller sub problems** which can be solved more easily. By separating the problem into sections, it becomes more feasible to manage and can be divided between a group of people according to the skill sets of different individuals.

This process requires software developers to consider a problems in terms of the **underlying sub problems that need to be solved** to achieve the desired result. 

For example in an adventure game we would have 
1. Characters
2. Adventures
3. Enemies 


These would then be broken down further to 
1. Characters
    a)  Characters' interactions
	b)   Characters' appearance
2.  Adventures 
	 a) Levels 
	 b) Backgrounds and settings
3.   Enemies 
	 a) Enemies' interaction
	 b) Enemies' appearance


Problems are commonly decomposed using top-down design, shown below.

This is also known as stepwise refinement, and is the preferred method used to approach very problems, as it breaks problems down into levels. Higher levels provide an overview of a problems , while lower levels specify in the detail the components of this problem.

The aim of using top-down design to to keep spiting problems into subproblem can repented as a **single task** and ideally in a self-contained module or sub routine. Each task can then be solved and developed as a subroutine by a different person. Once programmed, subroutines can also be **tested separately**, before being brought  together  and finally integrated.



## Identify the Components of a Solution

This is the stage in which the details about how each component is implemented are considered. 

In the same way that we broke down the problem, we must also build up to its solution. In order to identify the components of the solution, each programmer must evaluate the component of the problem allocated to them and assess how it can best be solved.


For example in out previous  decomposition for the book reservation system we would need to consider the **lowest level components**.

So on and so forth.

During this stage, its also useful to identify tasks which could be solved using an already existing module, subroutine or library. Reducing the complexity of the development stage by picking up on sections in which reusable components can be used is another benefit of using top-down design. 

Just merge all the components to together to get the full working solution.
